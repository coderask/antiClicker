<objective>
Implement the `electron-store`-backed persistent settings module with a zod schema gate at the type boundary. This satisfies FND-03 ("App settings persist via `electron-store` outside version control"). Phase 0's schema is intentionally minimal — exactly two fields (`launchCount`, `googleMapsApiKey`) — because the load-bearing proof is *that persistence works at all*, not the richness of what's stored. Phase 4 extends the schema.

Purpose: Two-fold. (1) Lock down the persistence pattern (ESM-only electron-store@11, zod safeParse on load, clearInvalidConfig self-heal, no dual-source-of-truth between zod and JSON Schema). (2) Provide the `launchCount` smoke-test field that the FND-03 e2e (00-06) will read across two `_electron.launch()` invocations to prove the write survived a relaunch.

Output: `src/main/config-store.ts` (~30 lines) plus a vitest unit test that exercises the schema in isolation (no electron-store actually written to disk in the unit test — we mock or just test the pure schema).
</objective>

```typescript
import type Store from 'electron-store';

export type AppConfig = {
  launchCount: number;            // nonneg int, default 0; incremented on every initConfigStore()
  googleMapsApiKey: string | null; // default null in Phase 0; Phase 4 will let users set it
};

export function initConfigStore(): Store<AppConfig>;
//   Idempotent: returns the same instance on repeat calls.
//   Side effect: increments launchCount on every call (FND-03 smoke proof).
//   On disk: ~/Library/Application Support/AntiClicker/config.json (macOS)

export function getStore(): Store<AppConfig>;
//   Throws if called before initConfigStore.

export const ConfigSchema: z.ZodSchema<AppConfig>;
//   Exported so unit tests + future code can validate raw input.
```
</interfaces>

<critical_pitfalls>
- PITFALL 3 from 00-RESEARCH.md: `electron-store@10+` is ESM-only. The `import Store from 'electron-store'` syntax requires `"type": "module"` in package.json (already set by 00-01). If a previous maintainer regresses to CJS, this file throws ERR_REQUIRE_ESM at runtime.
- PITFALL: Schema duplication. electron-store accepts its own JSON-Schema for validation. We choose zod-only as the source of truth and DO NOT pass a `schema` to the `Store` constructor — we validate via safeParse on load. This avoids two schemas drifting.
- PITFALL: Race with the e2e test. The FND-03 e2e launches Electron twice expecting `launchCount` to increment. If two test runs share a userData dir from a previous flaky run, the counts start from N>1. The test must read both counts and assert delta=1, not absolute values.
- PITFALL: `electron-store` reads `app.getPath('userData')` lazily — calling `new Store(...)` outside of `app.whenReady()` will throw. initConfigStore() must only be called from within whenReady (00-02's main entry does exactly this).
</critical_pitfalls>
</context>
