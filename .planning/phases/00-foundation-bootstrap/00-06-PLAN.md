<critical_pitfalls>
- PITFALL 7 from 00-RESEARCH.md: do NOT run the e2e against electron-vite dev — dev mode is always HTTP via the Vite dev server, so FND-02 would pass even if the production HTTP server were broken. The spec MUST `_electron.launch({ args: ['out/main/index.js'] })` against a freshly built `out/`. The `test:e2e` script must depend on `build`.
- PITFALL: persistence.spec.ts needs both `_electron.launch` invocations to share a userData dir, otherwise launchCount starts from 1 on both runs and the assertion is meaningless. Override the env (e.g., set HOME or XDG_DATA_HOME to a tmpdir) for both launches, OR pass an explicit `--user-data-dir` via args (Electron CLI flag). Cleanest: set the env once at top-of-test via `electron.launch({ env: { ...process.env, ELECTRON_USER_DATA: tmpDir } })` AND have config-store consume that env — but Phase 0 doesn't have that wiring. Use the env-override approach: set `HOME` (macOS/Linux) or `APPDATA` (Windows) to a fresh tmpdir before BOTH launches.
- PITFALL: `_electron.launch` doesn't auto-build. The test:e2e script must run build first.
- PITFALL: Playwright's Electron API uses `app.firstWindow()` to get the BrowserWindow's Page; calling `window.url()` returns the actual loaded URL (used for FND-02). Don't confuse with `window.location.href` (which requires evaluate).
</critical_pitfalls>
